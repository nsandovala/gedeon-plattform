## 3. Qué problema quedó abierto

### Formulario hace POST pero falla en persistencia

El formulario web envía datos correctamente a `POST /api/interest`.
El endpoint **recibe la petición** y llega al handler, pero **falla al hacer flush/insert** en PostgreSQL.

**Error principal actual:**
```
asyncpg.exceptions.ConnectionDoesNotExistError: connection was closed in the middle of operation
```

**Error previo resuelto:**
```
email-validator is not installed
```
→ Se agregó `email-validator` como dependencia (Pydantic EmailStr lo requiere).
Luego se quitó porque se hizo `email: str | None` en su lugar para simplificar.

### Docker
El `docker-compose.yml` es correcto, pero al ejecutar `docker compose up -d`
en Windows falló porque **Docker Desktop no estaba corriendo**:
```
unable to get image 'postgres:17-alpine': error during connect:
open //./pipe/dockerDesktopLinuxEngine: The system cannot find the file specified.
```

Docker CLI está instalado (v29.1.3) pero el daemon no se había iniciado.

### create_tables
`python -m scripts.create_tables` falló porque PostgreSQL no estaba accesible
en el momento de ejecución (contenedor no levantado).

## 6. Hipótesis de causa

No inventar soluciones mágicas. Hipótesis razonables:

1. **PostgreSQL no estaba listo al momento de crear tablas o insertar**
   - El contenedor puede tardar en inicializarse. El healthcheck está configurado pero no se esperó.
   - `create_tables` se ejecutó antes de que postgres aceptara conexiones.

2. **Problema de conexión async/asyncpg en db.py**
   - El engine se crea al importar el módulo (module-level). Si el primer request
     ocurre antes de que el pool esté listo, puede dar `ConnectionDoesNotExistError`.
   - Posible falta de `pool_size`, `max_overflow` o timeout configuration.
   - `get_db()` usa `async_session()` sin context manager de engine — podría necesitar
     verificación explícita de que la conexión está viva.

3. **Diferencia de entorno Windows vs Mac**
   - En Windows, Docker Desktop usa WSL2/Hyper-V. La resolución de `localhost:5432`
     puede comportarse diferente.
   - En Mac con Docker Desktop, `localhost:5432` debería funcionar igual.
   - El `DATABASE_URL` usa `localhost` — si en Mac el contenedor expone en otra
     interfaz, habría que ajustar.

4. **Posible problema con `db.flush()` sin transacción explícita**
   - `db.add(record)` + `await db.flush()` puede fallar si la sesión no tiene
     una transacción activa. En async SQLAlchemy, a veces es necesario
     `await session.begin()` explícito.

## 7. Próximo plan de debugging (Mac)

Pasos concretos, en orden:

```bash
# 1. Traer últimos cambios
git pull

# 2. Levantar PostgreSQL
docker compose up -d

# 3. Verificar que el contenedor corre
docker compose ps

# 4. Ver logs de postgres
docker compose logs postgres

# 5. Esperar healthcheck (5-10s) y verificar conexión
docker exec gedeon-postgres pg_isready -U gedeon -d gedeon

# 6. Instalar deps Python
cd apps/api
pip install -r requirements.txt

# 7. Crear tablas
python -m scripts.create_tables
# O: cd apps/api/scripts && python create_tables.py

# 8. Levantar API
cd apps/api
uvicorn app.main:app --reload --port 8000

# 9. Probar health
curl http://localhost:8000/health

# 10. Probar docs (abrir en browser)
open http://localhost:8000/docs

# 11. Probar POST manualmente
curl -X POST http://localhost:8000/api/interest \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Test User",
    "discord": "testuser#1234",
    "email": "test@example.com",
    "preferredMode": "competitive",
    "availability": "flexible",
    "message": "Handoff test"
  }'

# 12. Probar GET
curl http://localhost:8000/api/interest

# 13. Si POST falla → revisar db.py (ver hipótesis #2 y #4)
# 14. Si POST funciona → levantar web y probar form completo
cd apps/web && npm run dev
```

**Si vuelve a fallar en db.py:**
- Agregar logging explícito en `get_db()` para ver el estado de la sesión
- Probar sync alternativo: `await db.commit()` en vez de `await db.flush()`
- Considerar agregar `await engine.dispose()` al shutdown del app
